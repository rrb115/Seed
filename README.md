# Seed: Deterministic Workflow Completion Engine (DWCE)


The primary integration surface is the Go backend in `backend/`.
The `frontend/` folder is a reference client runtime and test surface for browser/offline behavior.

## Backend focus

The backend provides deterministic workflow and sync APIs for web applications.

- `GET /v1/manifest`
  - Returns signed workflow manifests.
  - Supports closure selection by `goal`, explicit `steps`, and `resources`.
  - Includes server-defined safety class metadata (`SAFE`, `EVENTUAL`, `UNSAFE`).
  - Rejects unsafe offline manifest requests.
- `POST /v1/sync`
  - Accepts operation batches generated by clients while online/offline.
  - Applies server merge/validation logic.
  - Returns sync job metadata for asynchronous completion tracking.
- `GET /v1/sync/status`
  - Returns sync job state and completion result.
- `POST /v1/verify-resource`
  - Verifies resource integrity for workflow dependencies.

How this helps web applications:

- Defines a server contract for offline-first operation processing and reconciliation.
- Centralizes consistency checks for workflows that cannot be finalized locally.
- Allows clients to execute deterministic local state transitions and submit operations later.
- Provides verifiable manifests and resources so clients can enforce integrity before execution.
- Separates browser runtime concerns from authoritative server state transitions.

## Backend implementation

- Implements HTTP API handlers and the sync/merge engine in `backend/`.
- Server entrypoint: `backend/cmd/server/main.go`.
- API routing and manifest logic: `backend/internal/api/server.go`.
- Sync and merge engine: `backend/internal/syncer/engine.go`.
- Security and manifest signing: `backend/internal/security/signer.go`.
- In-memory persistence and op store: `backend/internal/store/memory.go`.
- Test coverage for APIs and sync engine: `backend/internal/api/server_test.go`, `backend/internal/security/signer_test.go`, `backend/syncer/engine_test.go`.
- Runtime behavior implemented by the backend:
  - Deterministic manifest recomputation from server-side graph.
  - Safety evaluation and offline manifest rejection for `UNSAFE` workflows.
  - Idempotent apply of operation batches with operation ID deduplication.
  - Async sync job handling and status polling for long-running merges.

- Intended webapp integration points:
  - `GET /v1/manifest` to obtain signed workflow closures and resource lists.
  - `POST /v1/sync` to submit client-generated operation batches.
  - `GET /v1/sync/status` to poll async job state and results.
  - `POST /v1/verify-resource` to validate resource integrity prior to execution.

- Notes on runtime characteristics:
  - Current persistence is in-memory; durable stores and external workers are required for production-scale reliability.
  - Replace `memory` store and single-process queue with durable op logs, partitioned workers, and KMS-backed signing for production.

- See detailed architecture mapping: [docs/architecture.md](docs/architecture.md)

## What is implemented

- Frontend DWCE SDK with pluggable modules:
  - workflow engine (DAG/reachability)
  - workflow safety classifier (`SAFE`, `EVENTUAL`, `UNSAFE`)
  - dependency graph engine (step/resource maps + transitive closure)
  - manifest manager (signed manifest fetch/import)
  - offline state engine (deterministic local apply)
  - operation queue (durable IndexedDB queue)
  - sync agent (`POST /v1/sync`, async status polling)
  - service worker bridge
- Service Worker with:
  - app shell caching
  - manifest signature verification (Ed25519)
  - resource integrity verification (SHA-256)
  - offline fetch strategies
- Go backend services:
  - `GET /v1/manifest` with deterministic workflow closure support (`goal`, `steps`, `resources`)
  - server-side workflow safety metadata in signed manifests
  - unsafe workflow rejection for offline manifest requests
  - `POST /v1/sync`
  - `GET /v1/sync/status`
  - `POST /v1/verify-resource`

## Safety model

- `SAFE`:
  - Fully offline executable (user-owned state).
- `EVENTUAL`:
  - Offline executable but server validation on sync.
  - UX should treat results as provisional until validated.
- `UNSAFE`:
  - Requires strong global consistency (inventory, seats, balances).
  - Offline execution is blocked; DWCE stores draft intent and executes when online.

## SDK usage

```javascript
import { DWCE } from "./dwce/index.js";

await DWCE.init({ token: "dev-token" });

DWCE.registerWorkflow("note_draft", {
  steps: ["open_editor", "edit_content", "save_draft"],
  safety_class: "SAFE",
});

DWCE.registerWorkflow("support_ticket", {
  steps: ["fill_form", "attach_context", "submit_ticket"],
  requires_validation: true,
});

DWCE.registerStepDependencies("note_draft", {
  open_editor: ["/index.html", "/styles.css", "/app.js"],
  edit_content: ["/index.html", "/styles.css", "/app.js"],
  save_draft: ["/index.html", "/styles.css", "/app.js", "/offline.html"],
});

await DWCE.prepareWorkflow("note_draft");
await DWCE.queueOperation({
  workflow: "note_draft",
  object_id: "note:123",
  type: "set_field",
  path: ["content"],
  value: "Offline draft text",
});
await DWCE.sync();
```
